# 项目说明（CLAUDE.md）

> 这是一个 24 小时 GameJam 项目，单人开发，主题：**1min（一分钟）**。
> 开发以 vibe coding 为主。请优先保证“随时有一个能跑的版本”，而不是堆功能。

---

## 一、这是什么

一个**轻量网页小游戏**，核心是让玩家**用体感去估测“一分钟有多长”**。
不是显示一分钟，而是逼玩家凭身体感觉去判断时间。做得干净、有手感、一局几秒到一分钟就能体验完。

## 二、技术约束（重要，别违背）

- **单文件优先**：尽量都写在一个 `index.html` 里（HTML + CSS + JS 同文件）。文件实在太大再考虑拆分，拆分前先说一声。
- **纯原生**：用 vanilla JavaScript + Canvas 2D（或必要时直接用 DOM）。**不要引入任何框架或库**（不要 React / 不要 npm / 不要构建步骤），除非我明确同意。
- **零构建**：双击 `index.html` 用浏览器就能跑。不要引入需要编译/打包的方案。
- 用到图片/音频等本地资源时，告诉我可能需要起本地服务器（`python -m http.server`），不要默认本地 file:// 能加载一切。
- 音频注意浏览器策略：**首帧不要自动播声音**，等用户首次点击/按键后再启动。

## 三、核心玩法

**主题落地：让玩家“感受”一分钟到底有多长。**

- 玩家**按住**开始计时，**松手**结束计时。
- 目标：松手时的时长尽量接近 **60 秒**，但**不能超过**——超过即“爆掉”判负（类似 Price-is-Right 规则）。
- 越接近 60 秒（且不超）越成功，据此给分/评价。
- 结束后可一键重玩。

### 当前已实现的形态（main 分支）

- **本地多人（pass-and-play），1–4 人**：每人**按住一个键**、或**按住屏幕上的一个方块**加入；凑齐人数 → **3-2-1 倒计时** → 同时计时。
- 没爆掉且**最接近 60 秒者获胜**，结算显示各人时长与名次，可「再来一局」或「换人数」。
- 单人也走同一套流程（人数选 1）。

### 关键设计铁律（违背了游戏就废了）

- **计时过程中绝对不能显示正在走动的时间、秒数、或任何精确进度条。** 整个游戏的灵魂是逼玩家靠体感而非读数。可以有“正在计时中”的状态提示，但不能泄露具体时长。
- 结果**只在松手后揭晓**：显示实际时长、与 60 秒的差值、以及一个评价（如 完美 / 很接近 / 太短了 / 超时爆掉）。

### 关键实现注意

- **计时用时间戳，不要逐帧累加 dt**：按下记 `pressTime = performance.now()`，松手算 `elapsed = performance.now() - pressTime`（毫秒转秒）。这个游戏精度强相关，时间戳差值最准。
- **输入**：鼠标用 `pointerdown` 开始 / `pointerup` 结束；触摸用 pointer 事件已覆盖手机。
- **键盘要小心**：若支持空格键，**按住时 `keydown` 会重复触发**，必须用一个 flag 保证只有“第一次按下”才开始计时，`keyup` 才结束。否则会反复重置。
- 松手如果发生在窗口外（pointerup 丢失），考虑用 `pointercancel` / 失焦兜底，避免计时卡住。

### 后续扩展（先把上面基础做扎实，再逐个加；加之前跟我确认）

- **干扰玩家判断**（最有发挥空间的方向，**尚未做**）：计时过程中加入视觉/听觉干扰来打乱内心计数——移动/闪烁的元素、变速动画、误导性的假计时显示、节奏忽快忽慢的声音等。
- **多人**：本地轮流（pass-and-play）—— ✅ **已实现**（见上「当前已实现的形态」）。真正的**联网实时多人**仍属高风险，要做之前必须先跟我确认。
- **本地排行榜**：已在 `master` 分支实现一版（localStorage，单人：54–60 秒上榜、越接近越靠前、可选填名字），**尚未移植到 main 多人版**。想要可以从 master 接进来。

## 四、代码结构约定

游戏代码按这五块组织，保持清晰、加注释，方便我跟得上：

1. 画布与画笔（canvas + ctx），或 DOM 元素引用
2. 游戏状态（所有会变的数据集中放一起）。多人版用 `phase`（select → ready → countdown → playing → result）+ `slots`（固定槽位，每个 occupant 记录来源/按住/松手/爆掉/用时）来组织。
3. 输入（按住/松手：pointer + 键盘，注意上面的键盘重复触发问题）
4. 游戏循环（requestAnimationFrame：每帧 update → draw）。注：v1 计时本身是事件驱动的，循环里 update 可能很轻；但后续“干扰”功能会重度依赖这个循环，所以结构先留好。
5. 启动

## 五、给你的工作方式（agent 行为）

- **小步快跑**：一次只改一件事，改完简要说明改了什么、我该怎么验证。
- 较大改动先用 **plan 模式**给我看方案，我确认再动。
- **每到一个能跑的稳定版本，提醒我**（方便我用 checkpoint 留档/回退）。
- **保持简单**：能简单实现就别过度设计。这是 jam，不是上线产品。
- 不要擅自加依赖、加复杂结构、扩大 scope。拿不准就问我。
- 代码保持可读、有注释——我要看得懂自己的工程。

## 六、优先级（时间压力下的取舍）

1. 永远有一个**能跑、能玩完一局**的版本最重要。
2. 手感 / 即时反馈（按下与松手的反馈、结果揭晓的呈现、音效）优先于内容量。
3. 临近结束时**停止加功能**，只修 bug、调平衡、确保能正常提交。

## 七、已确认的决定

- **超时处理**：超过 60 秒**直接爆掉判负**（已定）。
- **胜负判定（多人）**：没爆掉且**最接近 60 秒者获胜**，结算按接近度排名（已定）。
- **单人评价区间**（仅 master 排行榜版用到）：差 ≤1s 完美 / ≤3s 很接近 / ≤6s 还行 / 更短「太短了」；上榜门槛 54–60 秒。
- **提交**：**GitHub 源码 + itch.io 可玩链接，两者都要**（已定）。
